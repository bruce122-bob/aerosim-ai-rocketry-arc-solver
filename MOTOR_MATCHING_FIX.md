# 🔧 发动机名称匹配修复

## 问题描述
用户报告：程序无法从 `.ork` 文件中识别 **F42-8T** 发动机，只能识别 F32 发动机。

## 根本原因
OpenRocket `.ork` 文件中的发动机命名格式多样化，而数据库匹配逻辑过于严格：

### 常见命名变体：
| .ork文件中 | 数据库中 | 之前匹配 | 现在匹配 |
|-----------|---------|---------|---------|
| `F42-8T` | `AeroTech F42T` | ❌ | ✅ |
| `F42T-8` | `AeroTech F42T` | ❌ | ✅ |
| `F42T` | `AeroTech F42T` | ✅ | ✅ |
| `AeroTech F42T-8` | `AeroTech F42T` | ✅ | ✅ |
| `F32T-6` | `AeroTech F32T` | ✅ | ✅ |
| `Estes F15-0` | `Estes F15-0,4,6,8` | ❌ | ✅ |

---

## 解决方案

### 新增智能匹配逻辑 (`orkParser.ts`)

```typescript
/**
 * Enhanced motor matching logic
 * Handles various naming formats
 */

// Step 1: 规范化名称
const normalizeMotorName = (name: string): string => {
  return name
    .toUpperCase()
    .replace(/AEROTECH|ESTES|CESARONI|APOGEE|QUEST/gi, '') // 移除制造商
    .replace(/\s+/g, '') // 移除空格
    .replace(/[-_]/g, '') // 移除分隔符
    .replace(/T(\d+)$/, 'T') // "F42T8" → "F42T"
    .replace(/(\d+)[A-Z]$/, '$1T') // "F428" → "F42T"
    .trim();
};
```

### 匹配步骤：

#### 1️⃣ **精确匹配**
```typescript
normalizedDesignation === normalizedDbName
```
- `"F42-8T"` → 规范化为 `"F42T"`
- `"AeroTech F42T"` → 规范化为 `"F42T"`
- ✅ 匹配成功

#### 2️⃣ **核心代码匹配**
```typescript
提取核心: /([A-M]\d{1,3}[A-Z]?)/i
```
- `"F42-8T"` → 核心 `"F42T"`
- 在数据库中搜索包含 `"F42T"` 的发动机
- ✅ 找到 `"AeroTech F42T"`

#### 3️⃣ **部分匹配（兜底）**
```typescript
使用 "-" 或空格分割，比较核心部分
```

## 技术细节

### 规范化规则：
1. 转换为大写
2. 移除制造商名称（AeroTech, Estes, Cesaroni等）
3. 移除空格和分隔符（`-`, `_`）
4. 统一延迟后缀格式（`F42T8` → `F42T`）

### 匹配优先级：
1. **精确匹配**（最高优先级）
2. **核心代码匹配**（使用正则提取）
3. **部分匹配**（兜底策略）

### 数据质量检查：
- 推力曲线点数 < 3
- 平均推力 < 5N
- F级以上发动机默认推力为10N（可疑）
- 推进剂质量或总质量过低（< 数据库的80%）

如果检测到低质量数据，自动从数据库获取高质量数据。
